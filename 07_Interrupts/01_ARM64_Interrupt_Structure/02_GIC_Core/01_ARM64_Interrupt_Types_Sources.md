Understanding **SGI, PPI, SPI, and LPI** is the key to understanding the entire GIC architecture.


> The interrupt type is determined by **who generates it** and **which CPU(s) should receive it**.

---

# Big Picture

```text
                CPU0      CPU1      CPU2
                  |          |         |
                  |          |         |
             +--------------------------------+
             |              GIC               |
             +--------------------------------+
                ^        ^         ^      ^
                |        |         |      |
               SGI      PPI       SPI    LPI
```

Each type exists because ARM had different interrupt delivery requirements.

---

# 1. SGI (Software Generated Interrupt)

```text
ID Range: 0-15
```

Generated by:

```text
Software
```

Not hardware.

Not UART.

Not Timer.

Not GPIO.

A CPU intentionally creates the interrupt.

---

# Why SGI exists

Suppose:

```text
CPU0
CPU1
CPU2
CPU3
```

are running Linux.

CPU0 modifies a page table.

Now CPU1, CPU2, CPU3 still have old TLB entries.

Need to tell them:

```text
Invalidate your TLB
```

How?

Send SGI.

---

## SGI Flow

### CPU0

Writes:

```asm
ICC_SGI1R_EL1
```

or older GICv2 software register.

Example:

```text
Send SGI 5 to CPU2
```

---

### GIC

Receives request.

```text
Interrupt ID = 5
Target CPU = 2
```

Marks interrupt pending in CPU2.

---

### CPU2

Gets IRQ.

Reads:

```asm
ICC_IAR1_EL1
```

returns:

```text
5
```

Kernel knows:

```text
This is SGI #5
```

---

### Handler runs

Example:

```c
flush_tlb();
```

---

### EOI

```asm
ICC_EOIR1_EL1
```

Interrupt completed.

---

# Why SGIs are important

Without SGIs:

```text
CPU0 cannot quickly signal CPU1
```

Multiprocessor OSes depend heavily on SGIs.

Used for:

* TLB shootdown
* Scheduler wakeups
* CPU stop requests
* Rescheduling
* RCU synchronization
* Cache maintenance

---

# Linux Example

When one core wakes another:

```text
CPU0
  |
  +--> SGI
           |
           v
         CPU3
```

Linux does this constantly.

---

# SGI is basically

```text
CPU --> CPU interrupt
```

instead of:

```text
Device --> CPU interrupt
```

---

# 2. PPI (Private Peripheral Interrupt)

```text
ID Range: 16-31
```

Generated by:

```text
Hardware
```

but local to one CPU.

---

# Why PPI exists

Some hardware belongs to a specific CPU.

Example:

```text
CPU0 local timer
CPU1 local timer
CPU2 local timer
CPU3 local timer
```

Each CPU has its own timer.

When CPU0 timer expires:

```text
Only CPU0 should get interrupt
```

Not CPU1.

Not CPU2.

Not CPU3.

Therefore:

```text
Private Peripheral Interrupt
```

---

# Example: ARM Generic Timer

Most common PPI.

```text
PPI 30
```

Typically.

---

## Timer Flow

CPU0 programs timer.

```c
timer = current + 10ms;
```

---

Timer reaches value.

```text
Timer0 expired
```

---

Timer hardware asserts:

```text
PPI30
```

---

CPU0's Redistributor/GIC sees:

```text
PPI30 pending
```

---

CPU0 receives interrupt.

ISR runs.

```c
scheduler_tick();
```

---

# Important

Same interrupt ID can exist on every CPU.

Example:

```text
PPI30
```

exists on:

```text
CPU0
CPU1
CPU2
CPU3
```

simultaneously.

Because PPIs are private.

---

Visualization:

```text
CPU0 Timer --> PPI30 --> CPU0

CPU1 Timer --> PPI30 --> CPU1

CPU2 Timer --> PPI30 --> CPU2
```

Same ID.

Different CPUs.

---

# Other PPIs

Examples:

```text
Architected Timer
PMU Overflow
Watchdog

```

Operating system timing depends on it.

Examples:



# 4. LPI (Locality-specific Peripheral Interrupt)

Introduced:

```text
GICv3
```

for modern PCIe systems.

---

# Problem with SPI

SPI space:

```text
32 - 1019
```

Only about:

```text
988 interrupts
```

Not enough.

---

Imagine server:

```text
100 NVMe SSDs
50 NICs
GPUs
Accelerators
```

Each device may need:

```text
100s or 1000s
```

of interrupt vectors.

---

Need:

```text
Tens of thousands
```

interrupts.

---

SPI cannot scale.

Therefore:

```text
LPI
```

created.

---

# What generates LPI?

Usually:

```text
MSI/MSI-X
```

interrupts from PCIe devices.

---

# MSI Reminder

Old systems:

```text
Device
   |
IRQ wire
   |
CPU
```

Physical pin.

---

Modern PCIe:

No interrupt wire.

Instead:

```text
Memory write
```

called MSI.

Example:

```text
Write address X
Write data Y
```

---

This write means:

```text
Interrupt occurred
```

---

# Why ITS needed

GIC doesn't understand:

```text
PCIe DeviceID
MSI EventID
```

directly.

Need translation.

---

ITS does:

```text
(DeviceID, EventID)
           |
           v
          LPI
```

---

# Example

NIC generates:

```text
DeviceID = 25
EventID = 100
```

---

ITS lookup:

```text
25,100
   |
   v
LPI 50000
```

---

Now GIC understands:

```text
Interrupt 50000 pending
```

---

# LPI Flow

```text
PCIe Device
      |
 MSI write
      |
      v
     ITS
      |
      v
   LPI 50000
      |
      v
 Redistributor
      |
      v
    CPU
```

---

# Why called Locality-Specific?

Because LPIs are managed through the target CPU's Redistributor.

Interrupt can be localized to a CPU.

Very efficient for:

```text
Multi-queue NICs
NVMe queues
Storage controllers
```

---

# NVMe Example

Imagine:

```text
64 CPUs
```

NVMe creates:

```text
64 completion queues
```

Each queue:

```text
1 MSI-X vector
```

---

Mapping:

```text
Queue0 -> CPU0 -> LPI50000

Queue1 -> CPU1 -> LPI50001

Queue2 -> CPU2 -> LPI50002
```

---

Each CPU processes its own I/O.

No contention.

Excellent scalability.

# Complete Comparison

| Type | Source         | Generated By         | Visible To       |
| ---- | -------------- | -------------------- | ---------------- |
| SGI  | Software       | CPU                  | Other CPU(s)     |
| PPI  | Local Hardware | CPU-local peripheral | One CPU          |
| SPI  | Shared Device  | UART/GPIO/DMA/etc    | One or more CPUs |
| LPI  | MSI/MSI-X      | PCIe devices         | Target CPU       |

---

# Memory Trick

Think of the interrupt source:

```text
SGI  -> CPU → CPU

PPI  -> Local hardware → Local CPU

SPI  -> Shared device → Selected CPU

LPI  -> PCIe MSI → ITS → Redistributor → CPU
```

Or even shorter:

```text
SGI = "Hey CPU!"

PPI = "My timer expired"

SPI = "UART/Ethernet/USB needs service"

LPI = "PCIe device completed work"
```

That's the intuition ARM architects had when they created these four interrupt classes. Once you understand **who generates the interrupt and who owns it**, the SGI/PPI/SPI/LPI distinction becomes very natural.

To really understand the GIC (Generic Interrupt Controller), don't memorize the registers first. Think about **what problem the GIC solves**.

---

# Why do we need a GIC?

Imagine a modern ARM SoC:

* 8 CPU cores
* UART
* Timer
* Ethernet
* USB
* PCIe devices
* GPU
* DMA engines

All these devices can raise interrupts.

Questions arise:

1. Which CPU should receive the interrupt?
2. What if multiple interrupts occur simultaneously?
3. Which interrupt has higher priority?
4. How does the CPU know which interrupt occurred?
5. How do we support thousands of interrupts?

The **GIC** is the hardware block that solves all these problems.

Think of it as:

```text
Devices
   |
   v
+-----------+
|    GIC    |
+-----------+
   |
   v
 CPU(s)
```

The GIC sits between peripherals and CPUs.

# Evolution

## GICv2

```text
            +----------------+
UART ------>|                |
Timer ----->| Distributor    |
GPIO ------>|     GICD       |
DMA ------->|                |
            +----------------+
                    |
      -----------------------------
      |                           |
      v                           v

+-------------+           +-------------+
| GICC CPU0   |           | GICC CPU1   |
+-------------+           +-------------+
      |                         |
      v                         v
    CPU0                      CPU1
```

Components:

### Distributor (GICD)

Global block.

One per GIC.

Responsible for:

* interrupt routing
* enable/disable
* priorities
* trigger configuration

### CPU Interface (GICC)

One per CPU.

Responsible for:

* presenting interrupt to CPU
* acknowledging interrupt
* end of interrupt

# Interrupt Types

Understanding this is critical.

---

## SGI (Software Generated Interrupt)

Generated by software.

Used for CPU-to-CPU communication.

Example:

```text
CPU0 wants CPU1 to flush TLB
```

CPU0 sends SGI.

```text
CPU0 ---> GIC ---> CPU1
```

Interrupt IDs:

```text
0-15
```

---

## PPI (Private Peripheral Interrupt)

Private to one CPU.

Example:

```text
CPU0 local timer
CPU1 local timer
```

Each CPU has its own timer interrupt.

Interrupt IDs:

```text
16-31
```

---

## SPI (Shared Peripheral Interrupt)

Generated by external peripherals.

Examples:

```text
UART
Ethernet
USB
DMA
GPIO
```

Interrupt IDs:

```text
32+
```

These are shared across all CPUs.

Managed by Distributor.

---

# Example 1: UART Interrupt in GICv2

Suppose UART receives data.

---

## Step 1: Device raises interrupt

```text
UART RX data arrives
```

UART asserts:

```text
IRQ = 45
```

Signal goes to Distributor.

```text
UART
  |
  v
GICD
```

---

## Step 2: Distributor checks configuration

Distributor looks at:

### Is interrupt enabled?

```c
GICD_ISENABLER
```

If disabled:

```text
drop interrupt
```

---

### Priority

```c
GICD_IPRIORITYR
```

Example:

```text
IRQ45 priority = 20
```

---

### Target CPU

```c
GICD_ITARGETSR
```

Example:

```text
IRQ45 -> CPU1
```

---

### Trigger type

```c
GICD_ICFGR
```

Can be:

```text
Edge triggered
Level triggered
```

---

## Step 3: Distributor forwards interrupt

```text
Distributor
    |
    v
GICC CPU1
```

---

## Step 4: CPU Interface checks mask

CPU1's GICC checks:

```c
GICC_PMR
```

Priority Mask Register.

Example:

```text
PMR = 100
```

Interrupt priority:

```text
IRQ45 = 20
```

Allowed.

Interrupt is signaled to CPU.

---

## Step 5: CPU enters exception

Hardware automatically:

```text
Save PC
Save PSTATE
Switch to IRQ mode
Jump to vector table
```

CPU starts executing ISR.

```text
IRQ Handler
```

---

## Step 6: ISR acknowledges interrupt

Read:

```c
GICC_IAR
```

Interrupt Acknowledge Register.

```c
irq = readl(GICC_IAR);
```

Returns:

```text
45
```

Now CPU knows:

```text
UART interrupt
```

---

## Step 7: Driver handles device

Driver reads UART data.

```c
while(UART_RX_READY)
    read_byte();
```

Interrupt source cleared.

---

## Step 8: End Of Interrupt

Write:

```c
GICC_EOIR = 45;
```

Meaning:

```text
I finished servicing IRQ45
```

---

## Step 9: GIC can deliver next interrupt

Interrupt leaves:

```text
Active state
```

Next pending interrupt may be delivered.

---

# Interrupt State Machine

Every interrupt has states.

```text
Inactive
Pending
Active
Active+Pending
```

---

## Inactive

No interrupt.

```text
IRQ idle
```

---

## Pending

Interrupt occurred.

Not yet acknowledged.

```text
UART ---> GIC
```

---

## Active

CPU acknowledged.

ISR running.

```text
GICC_IAR read
```

---

## Active + Pending

Interrupt occurs again while ISR executing.

Example:

```text
UART interrupt
ISR running

New UART data arrives
```

State becomes:

```text
Active + Pending
```

After EOI:

```text
Pending
```

CPU receives it again.

---

# Why GICv2 had a problem

For 64-core systems:

```text
64 CPUs
64 GICC blocks
```

Distributor had to track too much per-CPU state.

Scalability issue.

Hence GICv3.

---

# GICv3 Architecture

```text
               +----------------+
Devices ------>| Distributor    |
               | GICD           |
               +----------------+
                        |
         --------------------------------
         |              |              |
         v              v              v

      +------+      +------+      +------+
      | GICR |      | GICR |      | GICR |
      +------+      +------+      +------+
         |             |             |
        CPU0          CPU1          CPU2
```

Major change:

```text
GICC removed
```

Replaced by:

```text
System registers
```

---

# Why remove GICC?

GICC was MMIO.

Access example:

```c
readl(GICC_IAR);
```

MMIO access is slow.

In GICv3:

```asm
mrs x0, ICC_IAR1_EL1
```

Direct CPU register access.

Much faster.

---

# Redistributor (GICR)

Most important new block.

Per CPU.

Example:

```text
CPU0 -> GICR0
CPU1 -> GICR1
CPU2 -> GICR2
```

Handles:

```text
SGIs
PPIs
Local state
```

instead of Distributor.

---

# Flow of SGI in GICv3

CPU0 sends SGI to CPU1.

---

## Step 1

CPU0 writes:

```asm
ICC_SGI1R_EL1
```

---

## Step 2

GIC generates SGI.

```text
Target CPU = CPU1
```

---

## Step 3

CPU1's Redistributor receives it.

```text
GICR1
```

---

## Step 4

CPU1 sees interrupt.

ISR runs.

---

## Step 5

Acknowledge

```asm
mrs x0, ICC_IAR1_EL1
```

---

## Step 6

EOI

```asm
msr ICC_EOIR1_EL1, x0
```

---

# What Distributor does in GICv3

Distributor now mainly manages:

```text
SPIs
```

Shared interrupts.

Examples:

```text
Ethernet
UART
USB
PCIe
DMA
```

---

# What Redistributor does

Handles:

```text
SGI
PPI
per-CPU interrupt state
sleep/wakeup
```

This scales much better.

For 256 CPUs:

```text
Distributor
    +
256 Redistributors
```

instead of giant centralized tracking.

# ITS (Interrupt Translation Service)

Needed for PCIe MSI/MSI-X.

Before MSI:

```text
Device -> physical interrupt wire
```

Not scalable.

---

Modern PCIe uses:

```text
Message Signaled Interrupts (MSI)
```

Interrupt becomes:

```text
Memory Write
```

Example:

```text
PCIe NIC
writes
0xFEE00000
```

instead of toggling a pin.

---

Problem:

```text
Thousands of devices
Thousands of vectors
```

Need translation.

ITS solves this.

---

```text
PCIe Device
      |
      v
    MSI Write
      |
      v
      ITS
      |
      v
      LPI
      |
      v
      GICR
      |
      v
      CPU
```

# LPI (Locality-specific Peripheral Interrupt)

Designed for massive interrupt counts.

Traditional SPI:

```text
~1020 interrupts
```

Not enough.

Datacenter server:

```text
100 PCIe devices
1000 queues/device
```

Need huge space.

LPI provides:

```text
Millions of interrupt IDs
```

implemented using memory tables.


### 1. SSD generates MSI

```text
PCIe memory write
```

---

### 2. ITS receives MSI

```text
DeviceID
EventID
```

---

### 3. ITS lookup

Translation tables:

```text
DeviceID -> LPI
```

---

### 4. ITS generates LPI

```text
LPI 50000
```

---

### 5. Redistributor gets LPI

```text
GICR for CPU3
```

---

### 6. Priority check

```text
Interrupt priority

```

Compared.

### 7. CPU interrupt signal

```text
IRQ line asserted
```

---

### 8. CPU exception entry

```text
Vector table
```

---

```text
LPI 50000
```


---

### 11. End interrupt

```asm
msr ICC_EOIR1_EL1, x0
```

---

```text
Inactive
```

Ready for next interrupt.

# One-Line Mental Model

**GICD decides *whether and where* a shared interrupt should go, GICR manages *per-CPU interrupt state*, ITS converts PCIe MSI messages into LPIs, and the CPU interface/system registers allow the CPU to acknowledge and complete interrupt handling.**

End-to-end path in a modern ARM server:

```text
Device
  ↓
GICD (routing/config)
  ↓
ITS (for MSI only)
  ↓
GICR (per CPU)
  ↓
CPU IRQ exception
  ↓
ICC_IAR1_EL1 (ack)
  ↓
ISR executes
  ↓
ICC_EOIR1_EL1 (EOI)
  ↓
Interrupt completed
```

This is the complete lifecycle of an interrupt from the instant hardware raises it until the CPU finishes servicing it.


Understanding **SGI, PPI, SPI, and LPI** is the key to understanding the entire GIC architecture.

A useful way to think about them is:

> The interrupt type is determined by **who generates it** and **which CPU(s) should receive it**.

# 4. LPI (Locality-specific Peripheral Interrupt)

Introduced:

```text
GICv3
```

for modern PCIe systems.

---

# Problem with SPI

SPI space:

```text
32 - 1019
```

Only about:

```text
988 interrupts
```

Not enough.

---

Imagine server:

```text
100 NVMe SSDs
50 NICs
GPUs
Accelerators
```

Each device may need:

```text
100s or 1000s
```

of interrupt vectors.

---

Need:

```text
Tens of thousands
```

interrupts.

---

SPI cannot scale.

Therefore:

```text
LPI
```

created.

---

# What generates LPI?

Usually:

```text
MSI/MSI-X
```

interrupts from PCIe devices.

---

# MSI Reminder

Old systems:

```text
Device
   |
IRQ wire
   |
CPU
```

Physical pin.

---

Modern PCIe:

No interrupt wire.

Instead:

```text
Memory write
```

called MSI.

Example:

```text
Write address X
Write data Y
```

---

This write means:

```text
Interrupt occurred
```

---

# Why ITS needed

GIC doesn't understand:

```text
PCIe DeviceID
MSI EventID
```

directly.

Need translation.

---

ITS does:

```text
(DeviceID, EventID)
           |
           v
          LPI
```

---

# Example

NIC generates:

```text
DeviceID = 25
EventID = 100
```

---

ITS lookup:

```text
25,100
   |
   v
LPI 50000
```

---

Now GIC understands:

```text
Interrupt 50000 pending
```

---

# LPI Flow

```text
PCIe Device
      |
 MSI write
      |
      v
     ITS
      |
      v
   LPI 50000
      |
      v
 Redistributor
      |
      v
    CPU
```

---

# Why called Locality-Specific?

Because LPIs are managed through the target CPU's Redistributor.

Interrupt can be localized to a CPU.

Very efficient for:

```text
Multi-queue NICs
NVMe queues
Storage controllers
```

---

# NVMe Example

Imagine:

```text
64 CPUs
```

NVMe creates:

```text
64 completion queues
```

Each queue:

```text
1 MSI-X vector
```

---

Mapping:

```text
Queue0 -> CPU0 -> LPI50000

Queue1 -> CPU1 -> LPI50001

Queue2 -> CPU2 -> LPI50002
```

---

Each CPU processes its own I/O.

No contention.

Excellent scalability.

# Complete Comparison

| Type | Source         | Generated By         | Visible To       |
| ---- | -------------- | -------------------- | ---------------- |
| SGI  | Software       | CPU                  | Other CPU(s)     |
| PPI  | Local Hardware | CPU-local peripheral | One CPU          |
| SPI  | Shared Device  | UART/GPIO/DMA/etc    | One or more CPUs |
| LPI  | MSI/MSI-X      | PCIe devices         | Target CPU       |

---

# Memory Trick

Think of the interrupt source:

```text
SGI  -> CPU → CPU

PPI  -> Local hardware → Local CPU

SPI  -> Shared device → Selected CPU

LPI  -> PCIe MSI → ITS → Redistributor → CPU
```

Or even shorter:

```text
SGI = "Hey CPU!"

PPI = "My timer expired"

SPI = "UART/Ethernet/USB needs service"

LPI = "PCIe device completed work"
```

That's the intuition ARM architects had when they created these four interrupt classes. Once you understand **who generates the interrupt and who owns it**, the SGI/PPI/SPI/LPI distinction becomes very natural.

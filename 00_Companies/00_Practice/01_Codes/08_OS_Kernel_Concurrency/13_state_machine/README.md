# 13 — Finite State Machine (FSM)

## Problem
Implement a state machine: a finite set of states, events that drive transitions, optional entry/exit/transition actions. Used for protocols, parsers, UI flows, embedded controllers.

## Why It Matters
Anywhere reactive logic exists. Naive nested-if explodes; principled FSM tables stay readable and testable. Interviewers like FSMs because they expose your software-architecture instincts in a small space.

## Approaches

### Approach 1 — Switch-Case
```c
switch (state) {
case S_IDLE:
    if (ev == EV_START) { state = S_RUN;  on_start(); }
    break;
case S_RUN:
    if (ev == EV_STOP)  { state = S_IDLE; on_stop();  }
    break;
}
```
- Simplest, easiest to debug step-by-step.
- Scales badly past ~10 states / 20 events.
- Logic tangled with transitions — hard to visualise the machine.

### Approach 2 — Transition Table
Data-driven: a 2D table indexed by `(state, event)` → `(next_state, action_fn)`.
```text
typedef struct { state_t next; action_fn_t action; } cell_t;
cell_t table[NUM_STATES][NUM_EVENTS] = {
    [S_IDLE][EV_START] = { S_RUN,  on_start },
    [S_RUN] [EV_STOP]  = { S_IDLE, on_stop  },
    /* default cells: { S_INVALID, NULL } */
};

step(ev):
    c = table[state][ev]
    if c.action: c.action()
    state = c.next
```
- The machine is **visible as data** — print, validate, generate from spec.
- Memory grows as `S × E` — use sparse/list form for big alphabets.

### Approach 3 — Function-Pointer State (State Pattern in C)
Each state is a function `state_t (*handler)(event_t)`; returns the next state.
```text
typedef state_t (*handler_t)(event_t);
handler_t handlers[] = { idle_handler, run_handler, ... };

step(ev):
    state = handlers[state](ev)
```
- Localises per-state logic — easy to add states.
- Less table-y, more procedural; preferred when actions are heavy.

### Approach 4 — Hierarchical State Machine (HSM / Statechart)
States nested inside super-states; events unhandled by inner state propagate outward. Reduces transition duplication. Formalised by Harel statecharts; implemented in QP/Quantum frameworks.
- Powerful for complex UIs, telecoms.
- Add `entry`, `exit`, `do` activities.
- Heavier framework; usually overkill until > ~20 states.

### Approach 5 — Coroutine / Generator-Style (Protothreads / Duff's Device)
Linear-looking code with `await`-points that re-enter; under the hood it's a state machine generated by switch labels.
- Reads top-to-bottom — great for protocols (e.g. modem AT-command flows).
- Hides the FSM; harder to introspect.

### Approach 6 — Code Generation
Tools (Ragel, SMC, Spin) take a spec and emit one of the above forms. Best for huge, well-specified machines (regex engines, parsers).

## Comparison
| Approach | Add new state | Visualise | Memory | Speed | Best for |
|---|---|---|---|---|---|
| Switch-case | edit case | poor | small | fast | < 10 states |
| Transition table | edit table | excellent | S × E cells | one indirect call | medium FSMs |
| Function-ptr states | add fn | medium | small | one indirect call | per-state heavy logic |
| HSM | add nested state | excellent | larger | medium | UI / protocol stacks |
| Coroutine | add `await` | poor | small | fast | linear flows |
| Codegen | edit spec | excellent | depends | optimal | large fixed FSMs (regex) |

## Diagram (Mealy FSM)
```
    EV_START / on_start                 EV_STOP / on_stop
IDLE ─────────────────────► RUN ─────────────────────► IDLE
   \                                                   /
    \           EV_RESET / on_reset                   /
     ◄─────────────────────────────────────────────◄
```

## Key Insight
- The right representation depends on **what changes most often**: states (function-pointer), transitions (table), behaviour within a state (switch or coroutine).
- A table-driven FSM is **introspectable** — you can serialise, diff, and verify it offline. That's a huge win for protocols.
- HSMs eliminate the "every state needs to handle EV_CANCEL" boilerplate by putting CANCEL in a parent state.

## Pitfalls
- Forgetting an `(state, event)` cell → undefined behaviour. Pre-fill table with a safe default (log + stay).
- Side effects in transition guards that fire even when the guard returns false
- Re-entrant events (action triggers another event before transition completes) → queue events; process one at a time
- Mixing Mealy (outputs on transitions) and Moore (outputs on entry) inconsistently → confusing actions
- Coroutine FSMs use a `static` per-invocation state → not reentrant; not thread-safe
- Recording state in multiple variables (`bool is_running; bool is_paused;`) → invalid combinations; use a single enum
- No `entry`/`exit` hooks → cleanup forgotten on transitions

## Interview Tips
1. Sketch the state diagram before writing code.
2. Default to the **transition table** when there are > a few states — interviewers like data-driven design.
3. Mention `entry`/`exit` hooks; mention HSM if states share lots of common transitions.
4. State the invariant: "one state at a time, all transitions through the dispatcher".
5. Pre-fill table with a default cell that logs an unexpected event — production-grade.

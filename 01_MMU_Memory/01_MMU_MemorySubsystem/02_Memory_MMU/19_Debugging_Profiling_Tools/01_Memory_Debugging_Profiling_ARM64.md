**Category**: Debugging and Profiling Tools

## 1. /proc Filesystem Memory Debugging

```
/proc/pid/maps: virtual memory layout of a process
  Format: start-end perms offset dev ino pathname
  
  Example:
  7f8000000000-7f8000100000 r-xp 00000000 fd:01 12345678  /lib/libc.so.6
  7f8000100000-7f8000200000 r--p 00100000 fd:01 12345678  /lib/libc.so.6
  7f8000200000-7f8000201000 rw-p 00200000 fd:01 12345678  /lib/libc.so.6
  7f8001000000-7f8002000000 rw-p 00000000 00:00 0          [heap]
  
  Permissions: r=read, w=write, x=execute, p=private(COW), s=shared
  Offset: file offset (0 for anonymous)
  dev: device (00:00 for anonymous)
  
/proc/pid/smaps: extended VMA statistics (superset of maps)
  Each VMA has additional stats:
    Size:             VMA virtual size
    KernelPageSize:   actual page size (4KB, 2MB, 1GB)
    MMUPageSize:      MMU-level page size
    Rss:              resident set size (actually in physical memory)
    Pss:              proportional set size (shared pages divided by users)
    Shared_Clean:     clean pages shared with other processes
    Shared_Dirty:     dirty pages shared
    Private_Clean:    clean private pages
    Private_Dirty:    dirty private pages (will be written on fork)
    Referenced:       pages recently accessed
    Anonymous:        anonymous pages (not file-backed)
    Swap:             pages currently swapped out
    SwapPss:          proportional swap size
    Locked:           mlocked pages
  
  smaps_rollup: sum of all VMAs (single entry for whole process)
  
/proc/meminfo: system-wide memory statistics
  MemTotal:      total physical RAM
  MemFree:       completely free pages
  MemAvailable:  estimated available (includes reclaimable cache)
  Buffers:       buffer cache (block device)
  Cached:        page cache
  SwapCached:    recently swapped-in pages still in swap cache
  Active/Inactive: LRU list sizes
  AnonPages:     anonymous page memory
  Mapped:        file-backed pages that are mmap'd
  Shmem:         shared memory (tmpfs/shmem)
  KReclaimable:  kernel-reclaimable memory (slab cache)
  Slab:          kernel slab cache total
  SReclaimable:  reclaimable slab cache (page cache metadata)
  SUnreclaim:    non-reclaimable slab (kernel structures)
  VmallocTotal:  total vmalloc VA space
  VmallocUsed:   used vmalloc

/proc/vmstat: detailed memory event counters
  pgfault:        total minor page faults
  pgmajfault:     major page faults (required disk I/O)
  pgpgin/pgpgout: pages paged in/out to swap
  pswpin/pswpout: swap operations
  pgalloc_normal: buddy allocations from ZONE_NORMAL
  pgfree:         buddy frees
  pgreclaim_kswapd/direct: pages reclaimed
  compact_success/fail: memory compaction stats
  thp_fault_alloc: THP allocations
  thp_collapse_alloc: khugepaged collapses
```

## 2. KASAN (Kernel Address Sanitizer)

```
KASAN: kernel-level memory bug detector
  Detects: use-after-free, out-of-bounds, use-of-uninitialized
  
CONFIG_KASAN: enable KASAN
CONFIG_KASAN_GENERIC: software shadow memory (slower, more compatible)
CONFIG_KASAN_HW_TAGS: hardware MTE-based (ARM64-specific, ARMv8.5+)

Generic KASAN (software):
  Shadow memory: 1 byte per 8 bytes of kernel memory
  Shadow value:
    0: all 8 bytes valid/accessible
    1-7: only first N bytes valid (partial allocation at end)
    Negative (0xF0-0xFF): poisoned (various reasons)
      0xFB: slab redzone after object
      0xFC: slab redzone before object
      0xFD: freed slab object (use-after-free)
      0xFE: memory behind page
      0xFF: stack right redzone
  
  Instrumentation: compiler inserts checks before every access
    // Generated by compiler (kasan instrumentation):
    if (*(shadow_address(ptr)) != 0) kasan_report(ptr, size, is_write, ip);
  
  Performance: 2-4× slower, 2× more memory (shadow pages)
  
HW KASAN (ARM64 with MTE):
  Uses MTE hardware tag checking (see Cat14)
  No shadow memory needed
  Performance: ~1-2% overhead (hardware tag check is fast)
  Granule: 16 bytes (vs 8 bytes for generic)
  
  Allocation: heap_alloc():
    IRG X0, X0: random tag in X0[59:56]
    STG X0, [allocation]: stamp allocation tag into memory
    Return pointer with tag bits set
  
  Free: heap_free():
    IRG X0, X0: new random tag (different from allocation tag)
    STG X0, [freed_memory]: update tag to "freed" value
    Use-after-free: old pointer has old tag → MTE fault

KASAN report example:
  BUG: KASAN: use-after-free in process_data+0x12/0x34
  Read of size 8 at addr ffff888012345678 by task myprocess/1234
  
  Allocated by task 1234:
    kzalloc()
    my_alloc_func()
    ...
  
  Freed by task 1234:
    kfree()
    my_free_func()
    ...
  
  Stack trace shows exact allocation and free sites
  
kmemleak: different from KASAN
  Purpose: detect memory leaks (allocated but never freed)
  Method: scan all kernel memory for pointers to each allocation
          If no pointer found to an allocation → "leak"
  CONFIG_DEBUG_KMEMLEAK: enable
  /sys/kernel/debug/kmemleak: trigger scan, read report
  echo scan > /sys/kernel/debug/kmemleak
  cat /sys/kernel/debug/kmemleak: shows leaked objects
```

## 3. ARM64 PMU (Performance Monitoring Unit)

```
ARM64 PMU: hardware performance counters

PMU registers:
  PMCR_EL0: Performance Monitors Control Register
    E[0]: enable PMU
    P[1]: reset event counters
    C[2]: reset cycle counter (PMCCNTR_EL0)
    N[10:11]: number of event counters (read-only, typically 6-8)
  
  PMCNTENSET_EL0: enable specific counters
    Bit[31]: enable cycle counter (PMCCNTR_EL0)
    Bit[n]: enable event counter n (PMEVCNTRn_EL0)
  
  PMEVCNTRn_EL0: event counter n value (32-bit, wraps)
  PMEVTYPERn_EL0: event type for counter n
    Bits[15:0]: event type code
    U[16]: count in EL0 (user)
    K[17]: count in EL1 (kernel)
    NSK[18]: count in non-secure EL1
    M[24]: monitor mode
  
  PMCCNTR_EL0: cycle counter (64-bit)

ARM64 event codes (common):
  0x00: SW_INCR (software increment, not useful for profiling)
  0x01: L1I_CACHE_REFILL (L1 instruction cache miss)
  0x03: L1D_CACHE_REFILL (L1 data cache miss)
  0x04: L1D_CACHE (L1 data cache access)
  0x08: INST_RETIRED (instructions retired)
  0x09: EXC_TAKEN (exceptions taken — TLB refills, page faults)
  0x10: BR_MIS_PRED (branch mispredictions)
  0x13: MEM_ACCESS (memory accesses)
  0x14: L1I_CACHE (L1 instruction cache access)
  0x15: L1D_CACHE_WB (L1 data cache write-back)
  0x16: L2D_CACHE (L2 data cache access)
  0x17: L2D_CACHE_REFILL (L2 data cache miss)
  0x19: BUS_ACCESS (bus/coherency interconnect accesses)
  0x1A: MEMORY_ERROR (memory errors, ECC)
  0x1B: INST_SPEC (speculatively executed instructions)
  0x26: L1D_TLB_REFILL (L1 data TLB miss)
  0x27: L1I_TLB_REFILL (L1 instruction TLB miss)
  0x2D: L2D_TLB_REFILL (L2 TLB miss)
  0x37: L3D_CACHE_REFILL (L3 data cache miss)

Reading PMU from Linux:
  perf (user interface): wraps PMU access
  perf stat -e L1-dcache-misses,L2-cache-misses,cache-misses ./myapp
  perf stat -e arm_spe/mem_latency/ ./myapp  (SPE: Statistical Profiling Ext.)

ARM SPE (Statistical Profiling Extension):
  Unlike PMU: records actual sample records (address, latency, type)
  Purpose: find hot memory access patterns, latency outliers
  Data: PC, PA, VA, access type (load/store), latency, cache level
  perf mem record --call-graph dwarf ./myapp: ARM SPE-based profiling
  perf mem report: show memory access latency distribution
```

## 4. perf and ftrace for Memory

```
perf for memory profiling:

perf stat: high-level stats
  perf stat -e cycles,instructions,cache-misses,page-faults ./app
  Output:
    1,000,000 cycles
    500,000 instructions  # 0.5 instructions/cycle (IPC)
    10,000 cache-misses
    100 page-faults

perf mem: memory access profiling (requires SPE on ARM64)
  perf mem record ./app
  perf mem report --sort=mem,sym:
    Shows: load/store latency, cache levels, memory access hotspots

perf record + report: sampling profiler
  perf record -g -e cycles ./app: sample on cycles
  perf record -g -e LLC-load-misses ./app: sample on LLC miss
  perf report --stdio: show call graph

perf top: live sampling
  perf top -e L1-dcache-load-misses: show hot functions for L1 miss

ftrace for memory subsystem:
  /sys/kernel/debug/tracing/
  
  Tracing page fault paths:
    echo 'handle_mm_fault' > set_ftrace_filter
    echo function > current_tracer
    cat trace
  
  Tracing slab allocation:
    echo kmalloc kfree > set_ftrace_filter
    echo function > current_tracer
  
  Memory events (trace events):
    echo 1 > events/kmem/kmalloc/enable: trace all kmalloc calls
    echo 1 > events/kmem/kfree/enable: trace all kfree calls
    echo 1 > events/mm_vmscan/mm_vmscan_kswapd_wake/enable: kswapd events
    echo 1 > events/huge_memory/mm_collapse_huge_page/enable: THP collapses
    echo 1 > events/tlb/tlb_flush/enable: TLB flush events
    cat trace_pipe: live event stream

lockdep for memory locks:
  CONFIG_PROVE_LOCKING: enable lockdep
  lockdep: tracks lock acquisition order
  Detects: potential deadlocks, lock order violations
  Output in dmesg when potential deadlock detected:
    WARNING: possible circular locking dependency detected
    [lock A -> lock B -> lock A chain detected]
```

## 5. KASAN, UBSAN, and Crash Analysis

```
UBSAN (Undefined Behavior Sanitizer):
  CONFIG_UBSAN: enable
  Detects:
    - Integer overflow (signed)
    - Array index out of bounds
    - Invalid pointer dereference
    - Type mismatches
  
  UBSAN report:
    UBSAN: array-index-out-of-bounds in my_driver.c:123:3
    index 100 is out of range for type 'int [64]'

crash tool: analyze kernel crash dumps
  Prerequisites:
    CONFIG_CRASH_DUMP or CONFIG_PROC_VMCORE: enable crash dump
    kdump: capture second kernel that saves memory to vmcore
    crash-utils: user-space crash tool
  
  crash vmlinux /var/crash/vmcore:
    > bt           # backtrace of current/crashed task
    > log          # kernel messages (dmesg equivalent)
    > vm <pid>     # virtual memory info for process
    > files <pid>  # open files for process
    > kmem -s      # slab cache usage
    > kmem -f      # free list stats per zone
    > ps           # process list with memory info
    > struct mm_struct <addr>: dump mm_struct
    > rd <addr> 64: read memory at addr
    > dis <addr>:   disassemble

/proc/vmstat analysis for memory pressure:
  Watch these counters increase rapidly = problem:
    pgmajfault: major faults = swap thrashing
    pgscan_kswapd/direct: high = memory pressure
    pgsteal_kswapd: high = actively reclaiming
    compact_fail: high = fragmentation (huge page fail)
    oom_kill: OOM kills happening
  
  Script: watch changes:
    awk '/pgmajfault|pgscan|oom_kill/' /proc/vmstat

DAMON (Data Access Monitor):
  Linux 5.15+: lightweight memory access monitoring
  Purpose: profile "data access" patterns for memory optimization
  Uses: mark pages with PROT_NONE → fault on access → log access
  Interface:
    /sys/kernel/mm/damon/admin/: configure DAMON targets/schemes
  Use cases:
    proactive page reclaim (reclaim cold pages before pressure)
    page placement optimization
    NUMA migration hints
    CONFIG_DAMON, CONFIG_DAMON_VADDR (virtual address monitoring)
```

## 6. Interview Questions & Answers

**Q1: How do you find memory leaks in the Linux kernel?**
Use `kmemleak` (`CONFIG_DEBUG_KMEMLEAK`): periodically scans all kernel memory for pointers. Objects with no reachable pointer = leak candidates. Enable: at boot with `kmemleak=on`. Trigger: `echo scan > /sys/kernel/debug/kmemleak`. Read: `cat /sys/kernel/debug/kmemleak` shows leaked object addresses with allocation stack traces. For userspace: `valgrind --tool=memcheck`. For SLUB: `CONFIG_SLUB_DEBUG` + `/sys/kernel/slab/<name>/alloc_traces` shows allocation stacks.

**Q2: How does KASAN detect use-after-free bugs?**
KASAN poisons freed memory: after `kfree()`, the freed object's shadow byte is set to `0xFD` (poisoned-freed). Compiler-inserted instrumentation checks the shadow byte before every access. If shadow byte indicates poisoned: `kasan_report()` is called with the bad address, access type, and complete allocation/free stack traces. On ARM64 with MTE (`KASAN_HW_TAGS`): instead of shadow bytes, the allocation tag in the pointer is different from the tag stored in memory after free → hardware MTE fault on access.

**Q3: What ARM64 PMU events would you use to diagnose cache performance?**
For L1 data cache: `0x03` (L1D_CACHE_REFILL) and `0x04` (L1D_CACHE). Miss rate = refill/access. For L2: `0x17` (L2D_CACHE_REFILL) / `0x16` (L2D_CACHE). For TLB: `0x26` (L1D_TLB_REFILL) — high value = frequent TLB misses → consider huge pages. For branch prediction: `0x10` (BR_MIS_PRED). Using `perf stat -e cache-misses,cache-references,branch-misses`: exposes these via the perf abstraction layer on ARM64.

**Q4: What is /proc/pid/smaps and why is Pss useful?**
`smaps` shows detailed per-VMA memory statistics. `Pss` (Proportional Set Size) divides shared pages by the number of processes sharing them. Example: `libc.so` mapped by 100 processes — each process's Pss counts it as 1/100 × 2MB = 20KB. `Rss` would count the full 2MB for each. `Pss` gives a realistic per-process memory footprint. Sum of all processes' Pss ≈ total physical memory used. Useful for: container memory accounting, finding which process "really" uses the most memory.

**Q5: How do you use ftrace to trace memory allocation hotspots?**
1. Enable function tracing: `echo function > /sys/kernel/debug/tracing/current_tracer`. 2. Set filter: `echo kmalloc > /sys/kernel/debug/tracing/set_ftrace_filter`. 3. Or use trace events: `echo 1 > /sys/kernel/debug/tracing/events/kmem/kmalloc/enable`. 4. Clear and collect: `echo > trace; sleep 5; cat trace`. 5. For call graphs: `echo function_graph > current_tracer` gives indented call tree. For production: use `perf record -e kmem:kmalloc -g ./app; perf report` for sampling-based kmalloc profiling without full tracing overhead.

## 7. Quick Reference

| Tool | Purpose | When to Use |
|---|---|---|
| /proc/pid/smaps | Per-VMA memory stats | Analyze process memory breakdown |
| kmemleak | Kernel memory leak detection | Find kernel memory leaks |
| KASAN | Kernel memory bug detection | Use-after-free, buffer overflow |
| UBSAN | Undefined behavior detection | Integer overflow, type mismatches |
| perf mem | Memory access profiling | Find cache miss hotspots |
| ARM SPE | Hardware memory profiling | Latency + access pattern analysis |
| ftrace kmem | Allocation tracing | Find frequent/large allocations |
| crash | Post-mortem kernel analysis | Analyze kernel panic dumps |
| DAMON | Data access monitoring | Cold page detection, NUMA hints |
